# 一、项目功能与目的

```
完整落地搭建一个用于生成决策建议报告或AI知识类问答的agent+rag项目，这个项目是用于回复用户关于ai相关领域的问题，并且如果用户提出想要实现一个想法或者项目，agent将能够拆解用户的任务意图，把模糊的想法拆解成一个个可以具体的实现任务的步骤并返回总结告诉用户，ai领域的知识库将会有pdf、markdown、word类型的文件
```



# 二、想要在搭建中涉及的内容

基础功能：理解用户意图->多agent任务拆解/提取向量知识库中的知识->思考->工具调用->总结。
项目保障问题：
短期记忆与长期记忆如何联动？
不同工具调用时冲突怎么调用？
如何判断任务拆解是否错误、怎么回滚失败步骤？
多人同时并发会不会卡？
上下文窗口不够用怎么办？在任务中途记忆截断，那后续任务怎么办？
如何设计权限控制？
如何实现反馈循环学习机制？
AI Agent长期记忆系统设计不能仅靠简单RAG，核心是分层架构、双模存储、进化式读写与自我进化闭环。采用热-温-冷三层架构：热记忆存当前对话上下文，速度快容量小；温记忆是核心，用向量库+知识图谱双模存储长期偏好与事实，向量库负责“意会”联想，知识图谱负责“言传”推理；冷记忆存原始数据，容量大速度慢。两大核心流程是记录和检索：记录时LLM提取关键信息转向量和三元组，另一LLM做优化器合并/链接/新增记忆；检索时向量库语义搜索+知识图谱精确查询，结果融合给LLM。还有自我进化闭环，接收反馈→反思→提炼规则→后续遵循，实现优化。生产环境问题解法：记忆膨胀用进化机制精简，矛盾加规则版本号和优先级，高并发靠缓存+异步写入。
关于反思机制：发现逻辑冲突后，后续该怎么办？**1. 局部重试与自我修正（Self-Correction）**

- **触发场景：** 冲突较小，比如格式错误、代码缺少依赖、或者某个 Agent 漏掉了一个约束条件。
- **解决路径：** 反思 Agent 将“错误信息 + 原始任务 + 冲突原因”打包，**仅路由回发生错误的那个 Agent**。
- **执行逻辑：** 该 Agent 带着反思意见重新生成结果，然后再次进入反思环节。通常系统会设置一个**最大重试次数（Max Retries）**，防止死循环。

#### **2. 跨 Agent 协商与辩论（Debate / Negotiation）**

- **触发场景：** 两个 Agent 的输出存在事实或逻辑上的直接矛盾。例如：财务 Agent 说“预算只有 1 万”，但采购 Agent 却给出了“2 万”的采购方案。
- **解决路径：** 系统暂停当前流水线，开启一个**“辩论子图（Debate Subgraph）”**。
- **执行逻辑：** 将两个 Agent 的输出和各自的推理过程（Chain of Thought）同时暴露给对方，让它们互相指出对方的问题。如果经过 2-3 轮辩论仍无法达成一致，则触发降级机制（见第 3 点）。

#### **3. 全局回退与人工介入（Fallback & Human-in-the-loop）**

- **触发场景：** 冲突属于底层逻辑互斥，或者重试/辩论多次后依然无法解决。
- 解决路径：

  - **回退（Rollback）：** 状态机指针回退到上一个稳定的 Checkpoint，甚至回退到“任务拆解”阶段，让规划 Agent 重新制定计划。
  - **人工介入（Human-in-the-loop）：** 这是企业级应用中最重要的一环。系统挂起当前任务，将冲突点、各方观点打包发送给人类专家，由人类做出最终裁决，并将裁决结果作为新的上下文注入系统继续执行。

### **什么是真正的“Bad Case 闭环学习”？**

简单来说，Bad Case（坏例）就是AI在实际使用中“翻车”的记录。所谓的闭环，就是**把用户的每一次“不满意”，变成下一次发版的“养料”**。具体可以分为四步走：

**1. 怎么精准抓到 Bad Case？**
不能只靠人工去翻聊天记录，必须建立自动化的捕获机制：

- **用户显性反馈**：在对话界面设置“踩/差评/重新生成”按钮。用户只要点了这些，系统就自动把这条对话连同上下文打包标记为 Bad Case。
- **系统隐性监控**：通过代码硬规则自动抓错。比如AI回复为空、乱码、死循环、调用工具失败超过上限，或者触发了敏感词拦截，系统会直接将其判定为 Bad Case。
- **用户行为捕捉**：如果用户频繁重复提问、修改提问方式或者中途直接关掉窗口，这通常意味着AI没懂他的意思，这也是极佳的负反馈信号。

**2. 抓到了之后怎么处理？（核心：分类治理）**
不是所有的 Bad Case 都要拿去重新训练模型，必须对症下药：

- **事实错误/知识过期**：绝对不能让模型死记硬背，而是应该去修复底层的知识库（RAG检索增强）或更新数据库。
- **行为模式问题**：比如AI总是暴力拒答、废话太多或排版混乱。这时候需要人工写出标准的“优秀回答”，用这些数据去做针对性的微调（SFT），教模型学会正确的行为习惯。
- **风格偏好问题**：如果是语气太生硬，可以通过偏好优化（DPO）让模型学习什么是更好的表达方式，而不是改变它的知识储备。

**3. 如何防止同一个坑踩两次？**
建立一份固定的**“金标评测集”**。每次把修好的 Bad Case 强制加入这个测试题库中。以后无论AI怎么升级换代，上线前都必须跑一遍这个题库，确认以前犯过的错不会再犯（回归测试），这就相当于给产品建立了防错的“记忆”。

### **二、 为什么要做“灰度测试”？怎么做才不空泛？**

AI大模型具有不可预测性（比如偶尔会产生幻觉）。如果一上来就把新版本推给所有用户，万一系统崩溃或给出极其离谱的回答，后果不堪设想。灰度测试就是给新版本的AI穿上“安全带”。

**1. 灰度测试的具体节奏：**

- **影子模式评估**：新版本先在后台偷偷运行，对比它和老版本在面对同一句话时的决策差异，但不直接向用户展示结果，以此评估潜在风险。
- **小流量切分**：先只切 1%~5% 的真实用户流量给新版AI。在这期间，重点盯紧两个指标：一是 Token 消耗成本有没有异常飙升（防止AI陷入逻辑死循环烧钱），二是报错率和任务完成率有没有恶化。
- **逐步放大**：只有当小流量下各项指标都表现稳定，甚至优于旧版时，才会逐步扩大到全量发布。

**2. 兜底的“熔断机制”：**
在灰度期间，必须设定一条红线。比如，如果发现某段时间内AI的错误率突然上升了5%，或者平均响应时间翻倍，系统会自动触发“熔断”——瞬间切断新版本的权限，自动回滚到上一个稳定的旧版本，确保核心服务不中断。

**总结一下：**“抓好 Bad Case”就是建立一个**“线上采错 → 离线优化 → 评测验收 → 灰度上线”**的流水线；而“灰度测试”则是保证这条流水线产出的新零件不会让整台机器散架的安全阀。把这两套机制跑通，你的AI产品才算真正具备了自我进化的能力。
